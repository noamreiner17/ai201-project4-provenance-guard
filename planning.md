# Provenance Guard Planning

## Project Overview

Provenance Guard is a backend service that helps determine whether submitted text is likely AI-generated or human-written. Instead of making a simple yes/no decision, the system combines multiple detection signals to generate a confidence score and a transparency label. Every classification is stored in an audit log, and creators can appeal decisions they believe are incorrect.

---

# System Flow

### Submission Flow

1. User submits text through **POST /submit**.
2. The backend validates the request.
3. The submitted text is analyzed by the Groq LLM.
4. The same text is analyzed using Stylometric Analysis.
5. Both scores are combined into a final confidence score.
6. The confidence score determines the transparency label.
7. The classification is stored in the audit log.
8. The API returns the final response.

### Appeal Flow

If the creator disagrees with the classification:

1. Creator submits **POST /appeal**.
2. The system verifies the submission exists.
3. The creator's explanation is recorded.
4. The submission status changes to **Under Review**.
5. The appeal is stored in the audit log.
6. The API returns a confirmation response.

---

# Detection Signals

## Signal 1 – Groq LLM

### Measures

The language model evaluates:

* Overall writing style
* Sentence coherence
* Vocabulary and word choice
* Consistency of tone
* Common AI writing patterns

### Why

Large language models recognize semantic and stylistic patterns commonly found in AI-generated writing that are difficult to capture using simple statistical methods.

### Limitations

* Can misclassify polished human writing.
* Can be fooled by heavily edited AI-generated text.

---

## Signal 2 – Stylometric Analysis

### Measures

The system computes several measurable writing statistics:

* Sentence length variance
* Vocabulary diversity (Type-Token Ratio)
* Punctuation frequency

### Why

Human writing tends to be more varied, while AI-generated writing often follows more consistent structural patterns.

### Limitations

* Poems and song lyrics may produce misleading statistics.
* Very short text does not contain enough information.
* Technical writing may appear overly consistent despite being human-written.

# Features for Implementation

## Detection Signals

The system uses two independent detection signals.

Each signal produces a normalized score between **0.0** and **1.0**.

* **0.0 = Strongly Human**
* **1.0 = Strongly AI**

### Groq LLM

The Groq API analyzes:

* Writing style
* Sentence coherence
* Vocabulary
* AI writing patterns

The API returns a confidence score between **0.0** and **1.0**.

### Stylometric Analysis

The backend calculates:

* Sentence length variance
* Type-Token Ratio
* Punctuation frequency

Each metric is normalized to a value between **0.0** and **1.0**.

The Stylometric score is the average of the three metrics.

---

## Confidence Scoring

Both signals contribute equally.

```text
Final Confidence = (Groq Score + Stylometric Score) / 2
```

The resulting confidence score ranges from **0.0** to **1.0**.

---

## Uncertainty Representation

| Confidence Score | Attribution    | Transparency Label   |
| ---------------: | -------------- | -------------------- |
|      0.00 – 0.25 | `human`        | Likely Human-Written |
|      0.26 – 0.45 | `likely_human` | Likely Human-Written |
|      0.46 – 0.54 | `uncertain`    | Uncertain Result     |
|      0.55 – 0.75 | `likely_ai`    | Likely AI-Generated  |
|      0.76 – 1.00 | `ai`           | Likely AI-Generated  |

The **Uncertain** category is used whenever the two signals disagree significantly or neither signal provides enough confidence for a reliable decision.

Because false positives are more harmful than false negatives, the system prefers returning **Uncertain** rather than making an incorrect high-confidence prediction.

---

# Transparency Label Design

## High-Confidence AI

**Likely AI-Generated**

> Our system found strong evidence that this content was generated by artificial intelligence. Confidence: High.

---

## High-Confidence Human

**Likely Human-Written**

> Our system found strong evidence that this content was written by a human author. Confidence: High.

---

## Uncertain

**Uncertain Result**

> Our system could not confidently determine whether this content was AI-generated or human-written. This result should not be considered definitive and may require human review.

# Appeals Workflow

Only the creator who submitted the content can file an appeal.

The appeal request includes:

* `content_id`
* `creator_reasoning`

When an appeal is received, the system will:

1. Verify the submission exists.
2. Update the submission status to **Under Review**.
3. Store the creator's explanation.
4. Record the appeal in the audit log.
5. Return a confirmation response.

A reviewer will see:

* Content ID
* Creator ID
* Original submitted text
* Groq score
* Stylometric score
* Final confidence score
* Transparency label
* Creator's explanation
* Review status
* Timestamp

# Anticipated Edge Cases

## Poetry and Song Lyrics

Poems often contain repeated words, irregular punctuation, and short sentences. These characteristics may resemble AI-generated writing and increase false positives.

---

## Academic and Technical Writing

Research papers and technical documents often use formal language and consistent sentence structures. These characteristics may produce higher AI scores despite being written by humans.

---

## Very Short Text

Tweets, comments, or single-sentence responses provide too little information for reliable stylometric analysis, reducing confidence.

---

## Heavily Edited AI Content

If AI-generated content has been substantially rewritten by a human, the Groq model and Stylometric Analysis may disagree, resulting in an **Uncertain** classification.

---

## Non-Native English Writing

Writers who are not native English speakers may use simpler vocabulary or more consistent grammar, increasing the chance of a false positive.

### Architecture Narrative

When a creator submits text, the backend validates the request before sending it through both detection signals. The scores are combined into a final confidence score, which determines the transparency label shown to the user. The classification and supporting information are stored in the audit log before the API returns the response.

If the creator believes the result is incorrect, they may submit an appeal. The appeal changes the submission status to **Under Review**, records the creator's reasoning, stores the update in the audit log, and returns a confirmation response.
